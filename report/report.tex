页面整体为静态 HTML 与 CSS 实现，未引入复杂前端框架。布局上采用分区式结构，信息层级较清晰，并设置了响应式样式以适应常见屏幕尺寸。

\section{实现过程}

\subsection{使用的主要工具与技术}

\begin{itemize}
  \item \textbf{AI Agent：}主要使用 Codex，接入 GPT Plus 辅助阅读作业要求、生成项目文件、修改网页和整理报告。
  \item \textbf{网页技术：}使用 HTML 和 CSS 搭建静态个人主页、课程资料页和博客页。
  \item \textbf{版本管理：}使用 Git 管理修改历史，通过 commit 记录每一阶段工作。
  \item \textbf{部署方式：}使用 GitHub Pages 从仓库 \texttt{master} 分支根目录发布静态页面。
  \item \textbf{文件存储：}小文件直接放入仓库，中等文件使用 GitHub Release，较大视频和活动资料使用科大云盘外链。
\end{itemize}

\subsection{网页搭建过程}

首先，根据课程大作业要求建立个人主页项目，并用 \texttt{AGENTS.md} 固化项目规则，包括项目目标、文件结构、内容真实性要求、报告要求和验证清单。随后创建首页、博客页、课程资料页、样式文件和报告模板。

首页最初是较简单的占位版本，后来根据真实信息逐步调整为当前结构：个人简介、科研经历、课程与资料整理、博客列表。课程资料部分经历了多次迭代：最初在首页直接挂载资料，后来为避免首页过长，改为独立的 \texttt{course-materials.html}；再后来根据资料大小，将部分文件改为 GitHub Release 或科大云盘链接。

博客页面最初是占位草稿，后续替换为完整博客《AI Agent 辅助搭建个人主页实践与思考》，并逐步插入主页截图、\texttt{AGENTS.md} 截图、项目交接截图、提示词截图和文件存储结构图等材料。

\subsection{部署过程}

项目通过 GitHub Pages 部署。基本过程如下：

\begin{enumerate}
  \item 在 GitHub 上创建 \texttt{personal-homepage} 仓库。
  \item 将本地 Git 仓库与远程仓库关联，并执行 \texttt{git push -u origin master}。
  \item 在 GitHub 仓库的 \texttt{Settings -> Pages} 中选择从 \texttt{master} 分支的根目录发布。
  \item 等待 GitHub Pages 构建完成后，通过 \url{https://18446744.github.io/personal-homepage/} 访问主页。
\end{enumerate}

部署过程中曾遇到 GitHub Pages 分支选项显示 \texttt{None} 的情况，原因是远程仓库尚未成功收到本地提交；也遇到过 \texttt{RPC failed} 和网络中断问题。后续通过重新确认远程分支和重新推送解决，并在部署完成后检查主页、博客页、课程资料页和外部资源链接。

\subsection{遇到的问题及解决方法}

\begin{itemize}
  \item \textbf{提示词不够明确：}早期让 AI 自动整合个人信息时，出现信息重复、详略不当甚至过度扩写的问题。解决方式是重新按“模块--内容”整理需求，再要求 AI 按模块修改。
  \item \textbf{上下文延续问题：}项目多次跨会话修改，容易忘记已做决策。解决方式是维护 \texttt{AGENTS.md} 和 \texttt{docs/ai/handoff.md}，每次重要修改后记录当前状态、变更文件和后续任务。
  \item \textbf{资料文件体积差异大：}课程资料、小论文、PPT 和视频大小差异明显。解决方式是分级存储：小文件放仓库，中等文件放 GitHub Release，大文件使用科大云盘链接。
  \item \textbf{部署和推送问题：}推送远程仓库时曾出现 RPC 错误，GitHub Pages 初次配置时也因远程分支为空无法选择分支。解决方式是确认本地 commit、远程分支和实际仓库页面内容，再重新推送。
\end{itemize}

\section{AI Agent 使用说明}

\subsection{使用了哪些 AI 工具}

本次作业主要使用 Codex 作为命令行 AI Agent，并结合 GPT Plus 能力完成网页搭建、文件整理、Git 操作指导和报告撰写。没有采用单纯网页端问答加复制代码的方式，而是尽量围绕项目文件和 Git 历史进行完整工作流协作。

\subsection{AI 辅助完成了哪些内容}

\begin{itemize}
  \item 阅读并整理作业要求，明确需要完成个人主页、第一篇技术博客、报告和可访问链接。
  \item 生成并维护 \texttt{AGENTS.md}，明确项目规范、技术栈、内容规则和验证清单。
  \item 搭建静态网页结构，包括首页、博客页、课程资料页和统一样式。
  \item 协助整理课程资料和活动资料链接，给出 GitHub Pages、GitHub Release 和云盘分流方案。
  \item 指导 GitHub Pages 部署步骤，并分析推送失败、分支为空等问题。
  \item 协助将博客中的截图占位符替换为真实图片，并为图片添加统一样式。
  \item 协助完善本报告结构，使其覆盖作业要求中的主要要点。
\end{itemize}

\subsection{本人对 AI 生成内容做了哪些修改}

本人对 AI 生成内容进行了多轮人工修改和取舍：

\begin{itemize}
  \item 将主页内容替换为真实个人信息，包括院系、专业、兴趣方向、科研经历和联系方式。
  \item 调整主页模块，删除不需要的占位内容，增加课程资料、通识课程资料和课余活动资料链接。
  \item 手动修改部分 HTML 内容，如资源标签、页脚文字和博客段落，使其更符合个人表达。
  \item 对博客内容进行大幅重写，保留个人真实体验和反思，而不是直接采用 AI 生成的正式文风。
  \item 检查并整理文件存储方案，决定哪些文件放入仓库、哪些文件放入 Release 或云盘。
  \item 使用 Git 管理版本，在重要阶段提交修改，并通过历史记录找回早期页面版本用于对比。
\end{itemize}

\subsection{对 AI 辅助编程的思考}

本次实践说明 AI Agent 在项目初期搭建、文件结构整理、代码修改和问题排查方面具有较高效率，尤其适合帮助初学者快速理解一个完整网页项目的工作流。但 AI 的输出并不总是直接可用：当需求描述模糊时，AI 可能生成不符合预期的页面结构；当指令过于细碎时，效率又可能低于直接手动修改。因此，使用 AI Agent 的关键在于明确目标、给出边界、及时验收，并保留人工判断。

此外，AI Agent 与 Git、项目规范文档和 handoff 记录配合使用时效果更好。Git 负责保存版本，\texttt{AGENTS.md} 负责提供稳定规则，\texttt{handoff.md} 负责记录阶段性决策，三者共同降低了跨会话协作的混乱程度。
